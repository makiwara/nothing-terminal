# Cockpit UI — mockups and phone-UI specs

refs: `specs/ui/README.md` (the spec set), `specs/mockups/` (the rendered mockups), `specs/protocol.md` (the WS data plane, unchanged), `specs/control_plane.md` (the REST control plane, with the v1 additions the rework needs), `../nothing-to-say/journal/002/2026-06-24_36_terminal_cockpit.md` (the sibling design note, read-only), `android/` (the prototype this targets).

Built landscape mockups for the voice-driven terminal cockpit (`specs/mockups/terminal_cockpit.html`, `voice_review.html`, shared `styles.css` mirroring nothing-to-say's Pitch Cyan + PT Mono, rotated to a 427×240 landscape frame), then split implementation-grade UI specs out of them under `specs/ui/`: an index plus visual system, ring and screens, voice input, and after-send review. The mockups are the visual source of truth; each spec surface cites its card.

The design is authoritative for the phone UI and supersedes the sibling design note on four points, all in the mockups: no per-terminal ☰ menu (terminals carry only a small title; close/open moves to a dedicated ring manager page with Halt-each and Add-new); a two-stage voice flow (the reused recorder's own Send submits audio, then a separate review panel offers Cancel/Adjust/Confirm); an editable transcript (tapping it raises the system keyboard); and the inverted treatment (accent fill, black content) scoped to live capture and the management headers only.

The recorder surface was re-derived from nothing-to-say's `RecordingSurface.kt` rather than its `recording.html` mock, which diverges from the source: the capture block is inverted (accent fill, black dot/timer/waveform/buttons), the dot is static (not pulsing), the waveform animates, and the geometry is a top-pinned block with a hands-free area where the two button strips meet from opposite ends (Locked rises from below, Options drops from the top), so the active gesture windows are short (reveal 0.50→0.25, lock 0.25→0, options 0.50→0.75). After send the surface stays and pulsates, but quietly: the transcribing block is non-inverted (dark surface, accent content, a gentle content-only breath), because nothing is being captured.

The per-session WS data plane (`specs/protocol.md`) is unchanged; the design needs no new wire primitives. The REST control plane is now specced in `specs/control_plane.md` (split out of the root README): v0 as the stand-in implements it, plus the v1 additions the rework needs — richer `GET /sessions` metadata (`started_at`, `state`/`exit_code`) and exited-session retention for the manager page (a change from the current auto-drop in `session.go`), a display `command` on `GET /scripts` for the selector, and an optional refine `context` on `/voice` for Adjust. The action model is already tagged (`kind: text|signal`) and `send` already injects the client's text verbatim and statelessly, so the editable transcript needs no send-side change. The v1 fields are pending in both the stand-in and the nothing-serious backend. `android/README.md` keeps describing the shipped prototype and now points at `specs/ui/` for the target; the prototype rework remains pending.

Open decisions, recorded in the spec index: the Cancel colour uses nothing-to-say's `danger` token `#B68CFF` (a violet) provisionally, pending confirmation vs a literal magenta; the review-panel background is plain black provisionally; and whether the cockpit ships as its own app over a shared voice module or merges into nothing-to-say is an open architectural question that does not affect the UI spec.
